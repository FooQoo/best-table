# Unit of Work 分解（AWS AI-DLC 形式）

`docs/PLANS.md` のマイルストーンを、AWS AI-DLC（AI-Driven Development Life Cycle）の
Inception フェーズで生成される成果物の形式に合わせて Unit of Work（UoW）へ分解したものです。

構成は AI-DLC の `inception/application-design/` 配下の3成果物に対応しています。

- Unit of Work 定義（責務・スコープ）
- 依存関係マトリクス
- ストーリーマップ（`docs/PLANS.md` のタスク・受け入れ条件との対応）

各 UoW はさらに Bolt（数時間〜数日の短サイクル）に分解します。

各 Bolt は TDD（Red → Green → Verify）で進めます。テストコードを先に書いて失敗させ（Red）、
それを満たす実装をし（Green）、最後に Quality Gates で検証する（Verify）流れです。

**現在の進捗は [`docs/STATUS.md`](STATUS.md) で管理する。** 各 UoW が「未着手／計画済み／進行中／完了」の
どこにいるかは、このファイルではなく `docs/STATUS.md` を正とする。Bolt を1つ進めるごとに
`docs/STATUS.md` の該当行と「現在地」を更新する。

**各 UoW は着手前に必ず実装計画（`docs/plans/uow-N-plan.md`）を作成する。**
定義・依存関係・Bolt 分解が決まっていても、実際のコードベースの現状を踏まえた計画を書いてから
Red に入ることで、手戻りと見落としを防ぐ。テンプレートは [`docs/plans/TEMPLATE.md`](plans/TEMPLATE.md)、
各 UoW の計画は [`docs/plans/`](plans/) 配下を参照する。

## 0. 前提: テスト基盤の導入

現状 `package.json` にはテストランナーが未設定（Vitest 等なし）。TDD で進めるための前提として、
すべての UoW に先立って以下を導入する。

- **責務**: ユニットテスト実行基盤（Vitest + React Testing Library）を導入する。
- **依存**: なし
- **出力**: `pnpm test` で実行できるテストランナー、`app/**/*.test.ts(x)` を実行できる設定。
- **検証**: サンプルの1テストが `pnpm test` で green になること。

> 以降の Red ステップに書く `*.test.ts(x)` は、この基盤導入後に実行できる想定のテストコードです。

## 1. Unit of Work 定義

### UoW-1: プロダクト言葉遣い・状態の一貫性

- **責務**: トップ、ヒアリング、検索結果、比較の4画面で、文言・選択状態（相手種別・重視条件）の
  扱いを揃え、画面間の遷移を迷いなくする。
- **対象ルート**: `/`, `/hearing`, `/results`, `/compare`
- **スコープ外**: AI 評価ロジックの中身、データモデルの再設計。
- **入力**: 既存モック実装、`app/state/booking-context.tsx` の現状。
- **出力**: 揃った文言、画面間で一貫した状態受け渡し、迷わず比較まで到達できる導線。
- **元マイルストーン**: マイルストーン1（全タスク）。

### UoW-2: 比較フローの分かりやすさ・可逆性

- **責務**: 比較画面への遷移・離脱・戻りを分かりやすくし、空状態や境界状態（0件・1件・上限件数）
  で崩れないようにする。
- **対象ルート**: `/results`, `/compare`
- **依存**: UoW-1（状態受け渡しの一貫性が前提）
- **出力**: 戻れる比較フロー、崩れない空状態・境界状態表示。
- **元マイルストーン**: マイルストーン1（比較フロー・空状態タスク）。

### UoW-3: 相手種別に応じた評価の重み付け

- **責務**: 相手種別（重要顧客／初回取引先／社内上司）を AI 風評価の重み付けに反映し、
  画面上に直接見える形で強調する。
- **対象ルート**: `/results`, `/compare`
- **依存**: UoW-1
- **出力**: フローごとに異なる強調項目（個室・接客・格式 / 会話しやすさ・雰囲気・アクセス / 予算・落ち着き・使いやすさ）。
- **元マイルストーン**: マイルストーン2（重み付け・強調反映タスク）。

### UoW-4: 懸念タグ・推奨理由・AI 質問応答例

- **責務**: モック店舗に懸念タグと推奨理由を追加し、店舗詳細で想定される AI 質問応答例を用意する。
  懸念は画面上に直接表示し、ホバー依存にしない。
- **対象ルート**: `/stores/:storeId`, `/results`
- **依存**: UoW-1
- **出力**: 懸念タグ付きモック店舗データ、質問応答例、店舗詳細画面での表示。
- **元マイルストーン**: マイルストーン2（懸念タグ・推奨理由・Q&A タスク）。

### UoW-5: 最終候補の説明文面

- **責務**: 比較画面で選んだ最終候補について、理由と予約前の確認事項を含む説明文面を追加する。
- **対象ルート**: `/compare`
- **依存**: UoW-2, UoW-3
- **出力**: 最終候補ごとの推薦理由・確認事項の説明エリア。
- **元マイルストーン**: マイルストーン1（推薦理由エリア）＋マイルストーン2（最終候補向け文面）。

### UoW-6: ドメインデータモデルの再構成

- **責務**: 店舗データを `docs/MODEL.md` 準拠の単一 `Restaurant` 型として正式化し、根拠カテゴリ
  （口コミ・写真・席・メニュー・アクセス・店舗紹介文）、情報源を意識した懸念フィールド、
  `generatedAt`/`confidence` 等のメタデータを定義する。よく使う要約・スコアのキャッシュ方針を設計する。
  （着手時の実装計画（`docs/plans/uow-6-plan.md`）で判明: `docs/MODEL.md`/`docs/ARCHITECTURE.md` は
  すでに「生データと AI 派生評価を型として分離しない」という明示的な設計判断を記載済みだったため、
  当初この責務文にあった「分離する」は撤回済み。）
- **対象領域**: `app/mocks/`、ドメインモデル（`docs/MODEL.md` 準拠）
- **依存**: UoW-3, UoW-4, UoW-5（UI 側が要求するデータ形状が固まってから再構成する）
- **出力**: UI がそのまま描画できる店舗/評価データ構造、根拠カテゴリを示す AI コメント、
  データ不足を過信として見せない表現方針。
- **元マイルストーン**: マイルストーン3（全タスク）。

### UoW-7: 検索・空席・AI 生成の実接続

- **責務**: エリア／緯度経度／日時／人数／予算／ジャンル／個室条件の検索と空席情報ソースを接続し、
  店舗要約・スコアの事前生成またはキャッシュ、AI コメントの段階的表示、店舗詳細 Q&A の実データ接続を行う。
- **対象ルート**: `/results`, `/stores/:storeId`
- **依存**: UoW-6（データモデルが確定していること）
- **出力**: 通常検索と同等速度の結果表示、初期表示をブロックしない AI コメント、
  不確実性を明示する AI 出力。
- **元マイルストーン**: マイルストーン4（全タスク）。

## 4. Bolt 分解と TDD サイクル（Red → Green → Verify）

各 UoW は 1〜3 Bolt に分解し、Bolt 単位で Red（テスト先行）→ Green（実装）→ Verify（検証）を回す。
ロジックとして切り出せない純粋な UI/文言/操作感は Red 相当を「先に確認観点を書き出す」ことに読み替え、
Verify は手動確認とする。

共通 Verify（全 UoW 共通・`AGENTS.md` 準拠）:
- `pnpm run typecheck`
- `pnpm build`
- ルーティング/UI に影響する UoW では `/`, `/hearing`, `/results`, `/compare` を実機確認する。

### UoW-1: プロダクト言葉遣い・状態の一貫性

- Bolt 1-1（状態受け渡し）
  - Red: `app/state/booking-context.test.ts` に「相手種別を設定すると取得できる」
    「重視条件を更新すると `/hearing` → `/results` → `/compare` 相当の再取得でも保持される」
    テストを先に書き、未実装のため失敗させる。
  - Green: `app/state/booking-context.tsx` を実装し、テストを通す。
  - Verify: `pnpm test`（該当ファイル）+ 共通 Verify。
- Bolt 1-2（文言統一）
  - Red: 用語対応表（相手種別／重視条件の呼び方）を先にドキュメント化し、レビュー観点として確定する。
  - Green: トップ／ヒアリング／検索結果／比較の文言を対応表に合わせて修正する。
  - Verify: 4画面を一気通貫で操作し、詰まる導線・表記ゆれがないことを目視確認 + 共通 Verify。
- 完了の証拠: `pnpm test` のログ、一気通貫操作のスクリーンショット、共通 Verify のログ。

### UoW-2: 比較フローの分かりやすさ・可逆性

- Bolt 2-1（追加・削除・戻り導線）
  - Red: 比較対象の追加・削除ロジックに対するテストを先に書く（例: 追加で件数が増える、削除で減る、
    重複追加は無視される）。
  - Green: 比較追加・削除・戻り導線を実装する。
  - Verify: `pnpm test` + `/results` ⇄ `/compare` の往復操作を手動確認。
- Bolt 2-2（境界状態）
  - Red: 0件／1件／上限件数それぞれの入力に対して、期待する表示状態（空表示・通常表示・上限到達表示）
    を返すことをテストとして先に書く。
  - Green: 境界状態のハンドリングを実装する。
  - Verify: `pnpm test` + 3境界状態それぞれのスクリーンショットで表示崩れがないことを確認。
- 完了の証拠: `pnpm test` ログ、境界状態3パターンのスクリーンショット、共通 Verify のログ。

### UoW-3: 相手種別に応じた評価の重み付け

- Bolt 3-1（重み付けロジック）
  - Red: 相手種別（重要顧客／初回取引先／社内上司）を入力に、期待する強調項目を返す純粋関数の
    テストを先に書く（`app/utils/scoring.test.ts` 相当）。
    - 重要顧客 → 個室・接客・格式感
    - 初回取引先 → 会話しやすさ・雰囲気・アクセス
    - 社内上司 → 予算・落ち着き・使いやすさ
  - Green: 重み付け関数を実装しテストを通す。
  - Verify: `pnpm test`。
- Bolt 3-2（表示反映）
  - Red: カード／比較表コンポーネントが重み付け結果に応じた強調クラス・バッジを出すことを
    コンポーネントテストとして先に書く。
  - Green: カード・比較表への反映を実装する。
  - Verify: `pnpm test` + 相手種別3パターンの `/results` `/compare` を手動確認。
- 完了の証拠: `pnpm test` ログ、3パターンのスクリーンショット比較。

### UoW-4: 懸念タグ・推奨理由・AI 質問応答例

- Bolt 4-1（モックデータ）
  - Red: 店舗モックデータが懸念タグ・推奨理由フィールドを必ず持つことを検証するテストを先に書く
    （スキーマ/型レベルのテスト、または zod 等での必須項目チェック）。
  - Green: モック店舗データに懸念タグ・推奨理由を追加する。
  - Verify: `pnpm test`。
- Bolt 4-2（表示・Q&A例）
  - Red: 懸念タグがホバー操作なしで DOM 上に常時存在することを検証するコンポーネントテストを先に書く
    （ガードレール「懸念はホバーだけに隠さない」を担保）。
  - Green: 店舗詳細・カードの表示、AI 質問応答例を実装する。
  - Verify: `pnpm test` + `/results` `/stores/:storeId` を手動確認し、空席・予約成立・裏付けのない事実を
    断定していないか文面レビューする。
- 完了の証拠: `pnpm test` ログ、懸念タグ表示のスクリーンショット、Q&A 例のレビュー結果。

### UoW-5: 最終候補の説明文面

- Bolt 5-1（説明文生成）
  - Red: 最終候補と重視条件を入力に、「理由」と「予約前の確認事項」を含む構造化データ（または文字列）
    を返す関数のテストを先に書く。
  - Green: 説明文生成ロジックと `/compare` 上の表示を実装する。
  - Verify: `pnpm test` + `/compare` で最終候補選択後の表示を手動確認。
- 完了の証拠: `pnpm test` ログ、説明エリアのスクリーンショット。

### UoW-6: ドメインデータモデルの再構成

- Bolt 6-1（型・データ分離）
  - Red: 生の店舗情報型と AI 派生評価型が別型であること、既存 UI 用の変換関数が正しく合成できることを
    テストとして先に書く。
  - Green: `app/mocks/` の型・データを分離する。
  - Verify: `pnpm test` + `pnpm run typecheck`。
- Bolt 6-2（根拠カテゴリ・メタデータ）
  - Red: AI コメントが根拠カテゴリ（口コミ・写真・席・メニュー・アクセス・店舗紹介文）のいずれかを持つこと、
    `confidence`/`generatedAt` が必須であることをテストとして先に書く。
  - Green: フィールドを追加し、既存モックを更新する。
  - Verify: `pnpm test` + `/results` `/stores/:storeId` で根拠カテゴリ表示を手動確認。
- Bolt 6-3（キャッシュ方針）
  - Red: キャッシュキー生成・有効期限判定など方針の中核ロジックをテストとして先に書く。
  - Green: キャッシュ方針を実装する。
  - Verify: `pnpm test`。
- 完了の証拠: `pnpm test` ログ、型定義差分、根拠カテゴリ表示のスクリーンショット、データ不足ケースの表示確認結果。

### UoW-7: 検索・空席・AI 生成の実接続

- Bolt 7-1（検索条件連携）
  - Red: 検索条件（エリア／日時／人数／予算／ジャンル／個室）を入力に、想定するクエリ/リクエストを
    組み立てる関数のテストを先に書く。
  - Green: 実検索との連携を実装する。
  - Verify: `pnpm test` + `/results` の表示速度を手動計測（通常検索と同等であること）。
- Bolt 7-2（空席情報連携）
  - Red: 空席情報が未確定な場合に断定表現ではなく留保表現を返すことをテストとして先に書く。
  - Green: 空席情報ソースとの連携を実装する。
  - Verify: `pnpm test` + 未確定ケースの表示を手動確認。
- Bolt 7-3（AI コメント段階表示・Q&A 実接続）
  - Red: AI コメントの取得が初期表示をブロックしない（非同期に後から差し込まれる）ことを
    テストとして先に書く。
  - Green: 段階的表示と店舗詳細 Q&A の実データ接続を実装する。
  - Verify: `pnpm test` + AI コメント遅延表示・初期表示速度を手動確認。
- 完了の証拠: `pnpm test` ログ、初期表示速度の計測結果、不確実性表現のスクリーンショット。
