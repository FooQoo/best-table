# 計画

## 現在の目的

このサイクルで、Best Table の主要画面をスマホ幅でも破綻せず使える状態にする。

対象は `/`、`/hearing`、`/results` の全画面とする。特に `/results` は、店舗一覧、地図、店舗詳細パネル、比較サイドパネル、地図コンテキスト AI チャットが重なるため、デスクトップの左右分割をそのまま縮小するのではなく、スマホ幅では表示を切り替える体験にする。

ゴールは「スマホで最低限崩れない」だけではなく、店選びの流れがスマホでも自然に続くこと。トップで条件を入れ、ヒアリングを進め、検索結果で一覧と地図を切り替えながら詳細・比較・AI相談を使える状態を目指す。

## 決定事項

- `/`、`/hearing`、`/results` をすべて対象にする。
- `/results` のスマホ幅では、一覧と地図を同時に左右表示しない。上部の切替で「一覧」と「地図」を片方ずつ表示する。
- スマホ幅の店舗詳細パネル、比較サイドパネル、AI チャットは、狭い地図領域内に小さく重ねず、画面幅を使ったオーバーレイとして表示する。
- デスクトップ幅の既存体験（左一覧・右地図、地図内詳細パネル、地図内比較サイドパネル、地図内 AI チャット）は維持する。
- タブレット専用最適化、PWA、予約導線、検索 API の変更は対象外にする。
- ドメインモデル、AI 入力、セキュリティ境界は変更しない。

## 実装範囲

画面・パスの実装対象:

- トップ / 検索入口: `/`
- ヒアリング画面: `/hearing`
- 検索結果: `/results`
  - 店舗一覧
  - 地図
  - 店舗詳細パネル
  - 比較トレイ
  - 比較サイドパネル
  - 地図コンテキスト AI チャット
  - 条件編集パネル

実装対象外:

- タブレット専用最適化
- PWA 対応
- 予約導線の追加・変更
- 検索 API、AI API、Places API の契約変更
- 店舗データモデル、比較モデル、地図コンテキスト AI 相談モデルの変更
- 送客クリック計測

## プロダクトフロー計画

1. スマホで `/` を開き、エリア・日時・人数を入力する。
2. `/hearing` で相手、予算、重視条件を1ステップずつ入力する。
3. `/results` では最初に店舗一覧を見て、カードから詳細を開ける。
4. 地図を確認したい場合は「地図」表示へ切り替え、マーカーや「このエリアを検索」を使える。
5. 比較候補を2件以上追加すると、比較トレイから比較サイドパネルを開ける。
6. AI 相談は地図表示中でも、スマホ幅では画面幅を使ったチャットパネルとして開く。
7. どの表示でも、戻る・閉じる・比較操作が小さすぎたり画面外にはみ出したりしない。

## マイルストーン1: トップ・ヒアリングのスマホ表示

目的:

検索入口とヒアリング入力を、スマホ幅でも横スクロールや固定高の窮屈さなしに進められる状態にする。

タスク:

- `/` の検索条件入力カードをスマホ幅で縦積みにし、入力と CTA が画面幅に収まるようにする。
- `/` のヒーロー文言をスマホ幅で読み切れるサイズと余白に調整する。
- `/hearing` のステップカードをスマホ幅で画面高に依存しすぎない表示にする。
- `/hearing` の戻る/次へボタンをスマホ幅でも押しやすく配置する。

受け入れ条件:

- 375px 幅相当で `/` の入力部品が画面外にはみ出さない。
- 375px 幅相当で `/hearing` の各ステップが縦スクロールで確認でき、主要ボタンが画面外に固定されない。
- デスクトップ幅の既存レイアウトが大きく変わらない。

## マイルストーン2: 検索結果のスマホ表示切替

目的:

`/results` の左右分割レイアウトをスマホ幅で成立する形に変え、一覧と地図を切り替えて使えるようにする。

タスク:

- スマホ幅では「一覧」「地図」の切替を表示し、一度に片方だけを見せる。
- 初期表示は「一覧」とし、店舗カードの閲覧・比較追加・詳細表示を優先する。
- 地図表示では Google Maps、マーカー、マッチ度フィルタ、「このエリアを検索」、AI チャット導線が画面幅に収まるようにする。
- 検索条件サマリーと条件編集パネルをスマホ幅で折り返し・縦積みにする。
- 比較トレイをスマホ幅で縦積みにし、比較ボタンが押しやすい幅を持つようにする。

受け入れ条件:

- 375px 幅相当で `/results` の店舗一覧と地図が横並びのまま潰れない。
- 一覧表示中に店舗カードから詳細を開ける。
- 地図表示中に地図フィルタと AI チャット導線が画面外へ押し出されない。
- 比較候補の追加・比較パネルの開閉がスマホ幅でも使える。

## マイルストーン3: スマホ幅の詳細・比較・AI チャット

目的:

詳細、比較、AI チャットのオーバーレイをスマホ幅に合わせ、重要情報と閉じる操作が画面内に収まるようにする。

タスク:

- 店舗詳細パネルをスマホ幅では画面幅を使ったオーバーレイにする。
- 比較サイドパネルをスマホ幅では地図領域ではなく結果画面の幅いっぱいに表示し、比較表は横スクロールを許可する。
- AI チャットパネルをスマホ幅では画面幅を使ったパネルにし、入力欄と送信/中断ボタンが常に収まるようにする。
- オーバーレイを閉じても一覧/地図の表示状態が不自然に戻らないようにする。

受け入れ条件:

- 店舗詳細、比較、AI チャットの閉じるボタンがスマホ幅で見切れない。
- 比較表は列数が多くても横スクロールで確認でき、ページ全体を横スクロールさせない。
- AI チャット入力はスマホ幅でも1行初期表示から自然に拡張し、送信ボタンが見切れない。
- 店舗詳細パネルを開いたまま AI チャットを操作しても、既存方針どおり詳細パネルが不意に閉じない。

## 成功指標

ユーザー指標:

- スマホ幅でもトップから検索結果まで進める。
- 検索結果で一覧、地図、詳細、比較、AI 相談のどれかが画面外にはみ出して操作不能にならない。
- スマホで店選びを途中で諦める主因が、レイアウト崩れではなくなる。

開発指標:

- `pnpm test` / `pnpm run typecheck` / `pnpm build` が通る。
- `/`、`/hearing`、`/results` をスマホ幅とデスクトップ幅で確認する。
- `/results` では一覧/地図切替、店舗詳細、比較サイドパネル、地図コンテキスト AI チャットの開閉を確認する。

## モック UI 方針・レビュー記録

このサイクルは全画面のユーザー可視 UI を変更するため、`UNIT_OF_WORK.md` を生成する前に軽量な mock UI を実装して確認する。

mock UI は本実装の足場として残す想定で、実ルート上にスマホ幅の表示切替とオーバーレイ挙動を入れる。検索 API、AI API、Places API、ドメインモデルは変更せず、既存の mock mode / real mode のデータ形状をそのまま使って表示だけを検証する。

レビューで確認すること:

- `/` と `/hearing` がスマホ幅で横にはみ出さない。
- `/results` の一覧/地図切替が自然か。
- スマホ幅の詳細、比較、AI チャットが小さすぎず、閉じる操作が分かるか。
- デスクトップ幅の既存体験が崩れていないか。
