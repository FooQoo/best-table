# 計画

## 現在の目的

このサイクルで、Best Table のプロダクトゴールを次のとおり定義し、docs と実装に反映する。

> **Best Table のゴールは、接待・会食の店選びで「比較 → 納得」までを担い、納得した候補を予約手段へ送客すること。一休.comレストランに掲載されている店舗は一休の店舗詳細ページへ直接送客し、それ以外の店舗は Google Maps の場所ページへ送客する。予約・空席確認は送客先で完結させ、Best Table は予約機能を持たない。**

これまでのフローは「ヒアリング → 検索 → AI 評価 → 比較」で終わっており、比較した後にユーザーがどこへ進むのかが定義されていなかった。`mvp-cycle-4` で「最終候補の確定 → 予約導線受け渡し（`/reservation-handoff`）」というアプリ内の確定概念を撤回した判断は維持しつつ、その先の終点を「外部への送客リンク」として定義する。送客リンクは「この店に決める」という確定操作ではなく、「Google Mapで開く」と同列の外部確認導線として扱う。

このゴールは既存のガードレールと整合する。「空席・予約成立を断定しない」のは Best Table が予約の真実を持たないためであり、送客設計では「空席・予約は送客先で確認する」が公式の次アクションになる。

## 決定事項

- 候補の母集団は**限定しない**。既存の Places API (New) Text Search 起点のパイプラインを維持し、全候補を表示する（当初検討した「一休掲載店に限定する」案は、探索の幅を狭めるため撤回した）。
- 送客先は**ハイブリッド**とする。
  - 一休掲載店マスタと照合（店舗同定）できた店舗: 一休.comレストランの**店舗詳細ページへの直リンク**（「一休.comで空席を確認」）。URL はマスタに保持し、AI や文字列組み立てで生成しない。
  - 照合できなかった店舗: 既存の「Google Mapで空席・予約を確認」リンク（旧「Google Mapで開く」を改称）を送客導線として扱う（Google Maps の場所ページには、店舗によって Reserve with Google 経由の予約リンクが表示される）。
- 一休掲載店マスタは、このサイクルでは**モック fixture**（実在店の一休詳細ページ URL を少数手動記録したデータ）で用意し、repository パターンで将来の提携データ・API に差し替え可能な境界を切る。スクレイピングはしない。
- アプリ内の確定・決定概念（最終候補選択）は復活させない（`mvp-cycle-4` の撤回を維持）。
- 北極星指標は「検索セッションあたりの送客クリック率」とし、一休直リンク経由と Google Maps 経由を分けて定義する。このサイクルでは**指標の定義のみ**を行い、計測の実装（クリックイベントログ等）は含めない。

## 送客先データの調査メモ（2026-07 時点）

送客リンクの実データ化に向けて調査した結果。将来サイクルで real 実装する際の候補として記録する。

- **Google Places API (New) は予約プロバイダのリンクを返さない**。Google Maps の場所ページに表示される「席を予約」は Reserve with Google（一休・食べログ等のプロバイダ統合)によるユーザー向け画面の機能で、API のレスポンスには含まれない。API で取れるのは `googleMapsLinks.placeUri`（場所ページへのリンク。field mask 追加で SKU 影響の確認が必要）や `websiteUri`（公式サイト。Enterprise SKU）まで。`reservable`（boolean）はあるが URL は無い。現状の「Google Mapで開く」リンクは `placeId` から構築しており、追加コスト無しで場所ページへ送客できる。
- **一休掲載店マスタの公開 API は存在しない**。
- 一休.comレストランへの正式な送客経路は**バリューコマース経由のアフィリエイト提携のみ**。提携後は MyLink で店舗個別ページへのアフィリエイトリンクを作成できる。バリューコマースの Web Services（商品検索 API）で一休レストランの店舗データが取得できるかは提携後の確認事項。
- **YOLP（Yahoo!ローカルサーチ API）は提供継続中**（Client ID 登録で利用可）だが、レスポンスは Yahoo!ロコ系の POI 情報であり、一休.com の店舗詳細ページ URL は取得できない。LINEヤフーグループ内でも Yahoo!ロコの飲食予約は PayPayグルメ統合で、一休は別サービス。YOLP は店舗同定（電話番号・住所の照合補助）の候補には残るが、送客 URL の情報源にはならない。
- real 実装時の店舗同定キーは電話番号（正規化済み）を第一候補、店名+住所を補助とする。モックマスタでも同じ照合ロジックを通す。

## 実装範囲

画面・パスの実装対象:

- MAP 付き検索結果: `/results`（一休掲載店マスタとの照合、店舗カードの「一休.com掲載」表示）
- `/results` 内の店舗詳細パネル・比較サイドパネル（「一休.comで空席を確認」リンク。照合できた店舗のみ）
- 検索 resource route: `/api/restaurants/search/stream`・`/api/restaurants/search`（照合ステップの追加）

実装対象外:

- 実予約、実在庫連携（予約は送客先で完結する。Best Table は今後も予約機能を持たない）
- 送客クリックの計測実装（指標定義のみ。実装は将来サイクル）
- バリューコマース提携・アフィリエイトパラメータ付与（将来サイクル）
- 一休掲載店マスタの real データ接続（モック fixture のみ。repository 境界だけ用意する）
- 検索候補の母集団変更（一休掲載店への限定はしない。施設検索・AI 評価のパイプラインは全候補に対して現行どおり動く）
- アプリ内の最終候補確定・予約導線ページ（`mvp-cycle-4` の撤回を維持）
- 地図コンテキスト AI チャットの回答文言変更（「一休で確認」への誘導はプロンプト変更を伴うため将来サイクル）
- `googleMapsLinks.placeUri` / `websiteUri` の field mask 追加（SKU 影響が未確認のため。Google Maps 送客リンクは現行の `placeId` ベース URL を使い続ける）

## プロダクトフロー計画

1. ヒアリング入力: `/hearing`
2. 検索結果を取得: `/results`。施設検索（Places）で得た各候補を一休掲載店マスタと照合し、照合できた店舗に一休詳細ページ URL を付与する（候補の絞り込みはしない）
3. 店舗カードで一休掲載店には「一休.com掲載」の表示が付き、比較候補を選ぶ際の判断材料になる
4. 店舗カードから詳細パネルを開くと、一休掲載店には「一休.comで空席を確認」リンク、全店舗に「Google Mapで空席・予約を確認」リンクが表示される
5. 比較サイドパネルの各店舗列にも同じ導線が表示される
6. ユーザーはリンクから一休.comの店舗詳細ページ、または Google Maps の場所ページへ遷移し、空席確認・予約は送客先で行う（ここが送客 = プロダクトのゴール）

## マイルストーン1: 一休掲載店マスタと店舗同定

目的:

一休掲載店マスタ（モック fixture）と、Places 候補をマスタに照合する店舗同定の仕組みを作り、照合できた店舗に一休詳細ページ URL を付与する。

タスク:

- `IkyuReferral` 値オブジェクト（一休店舗詳細ページ URL、照合キー種別）と `Restaurant.ikyu` フィールド（`IkyuReferral | null`）を追加する。
- 一休掲載店マスタの repository（`app/server/repositories/`）を追加する。このサイクルではモック fixture を読むだけの実装とし、将来の提携データに差し替えられるインターフェースにする。
- 照合ロジック（placeId → 電話番号（正規化） → 店名+住所の優先順）を純粋関数として実装する。
- 検索パイプライン（`streamRestaurants`）で、施設検索の直後に照合を行い、`restaurant` イベントの時点で `ikyu` を付与する（AI 評価は待たない）。
- マスタの読み込みに失敗した場合は、検索を失敗させず全店舗 `ikyu: null`（一休リンク無し）で続行する。
- 照合できない店舗は除外せずそのまま表示する（`ikyu: null` が通常の状態）。

受け入れ条件:

- 一休掲載店マスタに照合できた店舗は `ikyu.url`（一休店舗詳細ページ URL）を持ち、照合できない店舗は `ikyu: null` のまま一覧・地図・比較に表示され続ける。
- 一休詳細ページ URL はマスタ由来の値だけが使われる（AI 生成・文字列組み立てによる URL が存在しない）。
- マスタ読み込み失敗時も検索・比較・ナビゲーションが使える。
- `pnpm test` / `pnpm run typecheck` / `pnpm build` が通る。

## マイルストーン2: 送客リンク UI

目的:

納得した候補から予約手段（一休.com または Google Maps）へ遷移できる送客導線を、確定概念を持ち込まずに表示する。

タスク:

- 店舗詳細パネルに、`ikyu` を持つ店舗だけ「一休.comで空席を確認」リンクを追加する（外部リンク。新しいタブで開く）。「Google Mapで空席・予約を確認」と同列の導線として配置する。
- 比較サイドパネルの各店舗列に同じリンクを追加する（`ikyu` を持つ店舗のみ）。
- 店舗カードに「一休.com掲載」の小さな表示を追加し、比較候補を選ぶ段階で直接送客できる店舗が分かるようにする。
- 文言は空席・予約成立を断定しないガードレールに合わせる（「確認」を促す表現。「予約する」「空席あり」とは書かない）。
- `ikyu` が無い店舗では一休リンクを表示しない（ダミーリンクや無効ボタンを置かない）。

受け入れ条件:

- 店舗詳細パネルと比較サイドパネルから、一休掲載店は一休.comの店舗詳細ページを新しいタブで開ける。
- 一休掲載店でない店舗は一休リンクが表示されず、「Google Mapで空席・予約を確認」だけが表示される。
- 「この店に決める」等の確定操作・最終候補パネルは存在しないままである。
- `pnpm test` / `pnpm run typecheck` / `pnpm build` が通る。

## 成功指標

北極星指標:

- 検索セッションあたりの送客クリック率（送客リンクのクリック数 ÷ 検索セッション数）。送客リンクは「一休.comで空席を確認」と「Google Mapで空席・予約を確認」の両方を数え、内訳（一休直リンク / Google Maps）を分けて見る。このサイクルでは定義のみで、計測は実装しない。

補助指標（同じく定義のみ）:

- 一休直リンク送客率（一休掲載店の表示回数に対する一休リンククリック率）
- 比較実施セッションのうち、比較後に送客クリックへ至った割合（比較経由の送客率）
- 送客クリックのうち比較サイドパネル発 / 詳細パネル発の内訳

ユーザー指標:

- 比較から予約行動（送客先）への導線が途切れず、次に何をすべきかが明確になる。
- 空席・予約の不確実性が「一休.com / Google Mapで確認する」という具体的な次アクションに変わる。

開発指標:

- 一休掲載店マスタが repository 境界で分離され、将来の提携データ・API に差し替え可能な状態になる。
- 送客 URL がマスタ由来（一休）と `placeId` ベース（Google Maps）に限定され、AI 生成・自由文字列由来の URL が存在しない。

## モック UI 方針

このサイクルは `/results` に新しい表示要素（一休送客リンク、店舗カードの「一休.com掲載」表示）を導入するため、`UNIT_OF_WORK.md` を生成する前に軽量な mock UI を実装して確認する。

mock UI は本実装の足場として残す想定で、店舗詳細パネル・比較サイドパネルへの「一休.comで空席を確認」リンクと、店舗カードの「一休.com掲載」表示を、モックデータ（`Restaurant.ikyu` を付与したテストデータ）で確認できるようにする。確定操作の UI は最初から含めない。mock UI 確認後、照合パイプラインの実装を含めて `UNIT_OF_WORK.md` を生成する。

mock UI レビューの記録:

- 初回レビューで「一休に拘らず、Google Map で予約ページのリンクを見たい」という指摘を受けた。Places API は予約プロバイダの URL を返さないため（上記調査メモ）、Google Maps の場所ページ経由を全店舗の予約到達手段として明示する方針にし、「Google Mapで開く」を「Google Mapで空席・予約を確認」に改称した。一休直リンク（照合できた店舗のみ）はより確実な送客先として維持する。
- mock UI は本実装の足場として残す。`Restaurant.ikyu` は mock UI 段階では optional で、照合パイプライン実装（UoW-1）で必須フィールド化する。
