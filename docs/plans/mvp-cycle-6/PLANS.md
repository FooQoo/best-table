# 計画

## 現在の目的

このサイクルで、Best Table のプロダクトゴールを次のとおり定義し、docs と実装に反映する。

> **Best Table のゴールは、接待・会食の店選びで「比較 → 納得」までを担い、納得した候補を一休.comの検索結果ページへ送客すること。予約・空席確認は一休.com側で完結させ、Best Table は予約機能を持たない。**

これまでのフローは「ヒアリング → 検索 → AI 評価 → 比較」で終わっており、比較した後にユーザーがどこへ進むのかが定義されていなかった。`mvp-cycle-4` で「最終候補の確定 → 予約導線受け渡し（`/reservation-handoff`）」というアプリ内の確定概念を撤回した判断は維持しつつ、その先の終点を「外部への送客リンク」として定義する。送客リンクは「この店に決める」という確定操作ではなく、「Google Mapで空席・予約を確認」と同列の外部確認導線として扱う。

このゴールは既存のガードレールと整合する。「空席・予約成立を断定しない」のは Best Table が予約の真実を持たないためであり、送客設計では「空席・予約は送客先で確認する」が公式の次アクションになる。

## 決定事項

- 候補の母集団は限定しない。既存の Places API (New) Text Search 起点のパイプラインを維持し、全候補を表示する。
- **一休掲載店マスタとの照合（店舗同定）は行わない。** 送客リンクは店舗名を検索語にした一休.comの検索結果ページ URL（`https://restaurant.ikyu.com/search?term=<店舗名>`）を、全店舗に一律で表示する。
  - 当初は一休掲載店マスタ（モック fixture）と Places 候補を照合し、一致した店舗だけに一休直リンクを付与する設計（UoW-1〜4）を実装したが、「一休に拘らず店舗名検索で十分」という判断のもと撤回した。照合ロジック・repository・マスタ fixture は削除し、`Restaurant` 型に送客専用フィールドは持たせない。
- 既存の「Google Mapで空席・予約を確認」リンク（`placeId` ベースで構築）は維持する。Google Maps の場所ページには店舗によって Reserve with Google 経由の予約リンクが表示される。
- アプリ内の確定・決定概念（最終候補選択）は復活させない（`mvp-cycle-4` の撤回を維持）。
- 北極星指標は「検索セッションあたりの送客クリック率」とする。このサイクルでは**指標の定義のみ**を行い、計測の実装（クリックイベントログ等）は含めない。

## 送客先データの調査メモ（2026-07 時点）

送客リンクの設計時に調査した内容。将来の要件変更時に参照する。

- **Google Places API (New) は予約プロバイダのリンクを返さない**。Google Maps の場所ページに表示される「席を予約」は Reserve with Google（一休・食べログ等のプロバイダ統合）によるユーザー向け画面の機能で、API のレスポンスには含まれない。API で取れるのは `googleMapsLinks.placeUri`（場所ページへのリンク。field mask 追加で SKU 影響の確認が必要）や `websiteUri`（公式サイト。Enterprise SKU）まで。`reservable`（boolean）はあるが URL は無い。既存の「Google Mapで空席・予約を確認」リンクは `placeId` から構築しており、追加コスト無しで場所ページへ送客できる。
- **一休掲載店マスタの公開 API は存在しない**。一休.comレストランへの正式な送客経路は**バリューコマース経由のアフィリエイト提携のみ**。
- **YOLP（Yahoo!ローカルサーチ API）は提供継続中**だが、レスポンスは Yahoo!ロコ系の POI 情報であり、一休.com の店舗詳細ページ URL は取得できない。
- 上記の理由により、店舗同定（照合）による直リンクは正確性が高い一方、マスタ整備・提携のコストが大きい。今回は「店舗名検索へのシンプルな送客」を優先し、店舗同定は行わない方針にした。

## 実装範囲

画面・パスの実装対象:

- `/results` 内の店舗詳細パネル・比較サイドパネル（「一休.comで空席を確認」リンク。全店舗に表示）

実装対象外:

- 実予約、実在庫連携（予約は送客先で完結する。Best Table は今後も予約機能を持たない）
- 送客クリックの計測実装（指標定義のみ。実装は将来サイクル）
- 一休掲載店マスタとの照合（店舗同定）。バリューコマース提携等による実データ化・直リンク化も含め将来サイクル
- 検索候補の母集団変更（施設検索・AI 評価のパイプラインは全候補に対して現行どおり動く）
- アプリ内の最終候補確定・予約導線ページ（`mvp-cycle-4` の撤回を維持）
- 地図コンテキスト AI チャットの回答文言変更（「一休で確認」への誘導はプロンプト変更を伴うため将来サイクル）

## プロダクトフロー計画

1. ヒアリング入力: `/hearing`
2. 検索結果を取得: `/results`（施設検索・AI評価は現行どおり。一休掲載店マスタとの照合は行わない）
3. 店舗カードから詳細パネルを開くと、全店舗に「一休.comで空席を確認」リンク（店舗名検索）と「Google Mapで空席・予約を確認」リンクが表示される
4. 比較サイドパネルの各店舗列にも同じ導線が表示される
5. ユーザーはリンクから一休.comの検索結果ページ、または Google Maps の場所ページへ遷移し、空席確認・予約は送客先で行う（ここが送客 = プロダクトのゴール）

## マイルストーン1: 一休.com検索リンクの送客導線

目的:

納得した候補から一休.com（店舗名検索）または Google Maps へ遷移できる送客導線を、確定概念を持ち込まずに全店舗へ表示する。

タスク:

- 店舗名から一休.comの検索結果ページ URL を組み立てる純粋関数 `buildIkyuSearchUrl`（`app/utils/`）を実装する。URL は `https://restaurant.ikyu.com/search?term=<店舗名>`（空白は `+` エンコード）。
- 店舗詳細パネルに「一休.comで空席を確認」リンクを追加する（外部リンク。新しいタブで開く）。既存の「Google Mapで空席・予約を確認」と同列の導線として配置する。
- 比較サイドパネルの各店舗列に同じリンクを追加する。
- 文言は空席・予約成立を断定しないガードレールに合わせる（「確認」を促す表現。「予約する」「空席あり」とは書かない）。
- 送客リンクは全店舗に一律表示する（店舗の状態による出し分けはしない）。

受け入れ条件:

- 店舗詳細パネルと比較サイドパネルから、全店舗が一休.comの検索結果ページ（店舗名で検索）を新しいタブで開ける。
- 一休.com URL は店舗名から機械的に組み立てた値のみが使われる（AI 生成・文字列組み立て以外の外部データ由来の URL が存在しない）。
- 「この店に決める」等の確定操作・最終候補パネルは存在しないままである。
- `pnpm test` / `pnpm run typecheck` / `pnpm build` が通る。

## 成功指標

北極星指標:

- 検索セッションあたりの送客クリック率（送客リンクのクリック数 ÷ 検索セッション数）。送客リンクは「一休.comで空席を確認」と「Google Mapで空席・予約を確認」の両方を数える。このサイクルでは定義のみで、計測は実装しない。

ユーザー指標:

- 比較から予約行動（送客先）への導線が途切れず、次に何をすべきかが明確になる。
- 空席・予約の不確実性が「一休.com / Google Mapで確認する」という具体的な次アクションに変わる。

開発指標:

- 送客 URL が店舗名ベース（一休.com検索）と `placeId` ベース（Google Maps）の機械的な組み立てに限定され、AI 生成・自由文字列由来の URL が存在しない。

## モック UI 方針・レビュー記録

このサイクルは `/results` に新しい表示要素（一休送客リンク）を導入するため、軽量な mock UI で確認しながら進めた。

- 初回レビューで「一休に拘らず、Google Map で予約ページのリンクを見たい」という指摘を受け、Google Maps の場所ページ経由を全店舗の予約到達手段として明示する方針にし、リンク文言を「Google Mapで開く」から「Google Mapで空席・予約を確認」に改称した。
- 続けて、一休掲載店マスタとの照合（店舗同定）パイプラインを実装したが（`IkyuListingRepository` / `matchIkyuListing` / `Restaurant.ikyu`）、レビューで「一休の検索結果ページへ飛ばすだけでよい」という判断があり、照合の仕組み一式を撤回した。送客リンクは店舗名を検索語にした一休.com検索URLへの単純なリダイレクトとし、全店舗に一律で表示する。
- URL のクエリ文字列は空白を `+` でエンコードする形式（`URLSearchParams` 相当）を採用した。
