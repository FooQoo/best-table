# 計画

## 現在の目的

`/results` の「比較する」ボタンがページ遷移（`/compare`）になっており、比較検討中に検索結果の地図・店舗一覧から離れてしまう。このサイクルでは、比較を `/results` 内のサイドパネルに一本化し、地図コンテキスト AI チャットと同じ配置ルールで、地図エリアを比較コンテンツで上書き表示する形に変える。

あわせて、比較表の「この店に決める」ボタンと、それによって表示される最終候補パネル（`FinalStorePanel`）を廃止する。この操作は `docs/DESIGN.md` / `docs/MODEL.md` に記載のある「最終候補選択 → 予約導線への遷移（`/reservation-handoff`、未実装）」という将来ロードマップの入口だったが、今回このロードマップ自体を撤回し、関連ドキュメントからも削除する。比較の役割は「複数店舗を同じ軸で見比べる」ことに絞り、決定・確定という概念は持たない。

## 実装範囲

画面・パスの実装対象:

- MAP 付き検索結果: `/results`
- `/results` 内の比較サイドパネル（地図エリアを上書き表示）
- 比較表（`CompareTable` 相当）の `/results` サイドパネルへの移設
- `/compare` ルートの廃止

実装対象外:

- 比較軸・評価ロジックの変更（`getEmphasisKeys` などは既存のまま流用）
- 実予約、実在庫連携
- 予約導線への遷移（`/reservation-handoff`）— ロードマップ自体を撤回するため、今回もその後も対象外
- AI チャット（`ResultsAiChat`）の挙動変更
- 店舗カード・店舗一覧・地図マーカーの表示項目変更

## プロダクトフロー計画

1. ヒアリング入力: `/hearing`
2. 検索結果を取得し、左に店舗一覧、右に地図を表示: `/results`
3. 店舗カードで比較に追加（2〜5件）
4. 画面下部の比較トレイで「比較する」を押す
5. 地図エリアが比較コンテンツで上書きされ、比較表が表示される
6. ユーザーは比較表を見ながら店舗を見比べる（最終候補の確定操作はない）
7. 「閉じる」で地図表示に戻り、店舗一覧・比較追加を続けられる

## マイルストーン1: 比較のサイドパネル化

目的:

比較体験を `/compare` へのページ遷移から、`/results` 内で完結するサイドパネルに変える。地図コンテキスト AI チャットと同じ配置ルール（地図エリア内に重ねる）に揃えつつ、比較表は店舗数に応じた幅が必要なため、AI チャットの細い右スライド幅とは異なり地図エリア全体を上書きする。

タスク:

- `CompareTray` の「比較する」ボタンを `/compare` への `navigate` から、比較サイドパネルの開閉トグルに変更する。専用の閉じるボタン（×）は置かず、同じボタンを押すたびに開閉し、開いている間はボタン文言を「比較を閉じる」に変える。
- 比較サイドパネルは `/results` の地図エリア（`ResultsMap` を包む `relative` コンテナ）内に配置し、開いたときに地図エリアを隙間なく完全に上書きする。
- サイドパネルの中身は既存の `CompareTable` 相当（店舗名、ジャンル、エリア、予算、個室・席、代表写真、個室有無、AI によるマッチングポイント要約、比較軸の重視表示）を移設する。
- 「この店に決める」ボタン、選択状態、`FinalStorePanel` は移設せず廃止する。
- サイドパネルを開いても、左の店舗一覧・比較トレイの操作は維持する。
- `/compare` ルート（`app/routes/_layout.compare.tsx`）と `CompareScreen`、`EmptyCompareState` を削除する。
- `state.finalStoreId` / `selectFinalStore`（`app/state/booking-context.tsx`）を state・型定義から削除する。
- `FinalStorePanel`、`buildFinalStoreMessage`（`app/utils/final-candidate-message.ts`）、`buildGoogleMapsUrl`（`app/utils/google-maps-url.ts`）とそれぞれのテストを削除する（他に利用箇所がないため）。

受け入れ条件:

- `/results` の比較トレイで2〜5件を選び「比較する」を押すと、ページ遷移せずに地図エリアが比較コンテンツで上書きされる。
- 比較コンテンツに店舗名・ジャンル・エリア・予算・個室・代表写真・懸念点・比較軸の重視表示が表示される。
- 比較コンテンツに「この店に決める」ボタンや最終候補パネルは表示されない。
- 開いている間に同じボタン（「比較を閉じる」）をもう一度押すと地図表示に戻り、店舗一覧・比較トレイの操作がそのまま続けられる。
- `/compare` という URL・ルートがアプリに存在しない。
- `pnpm run typecheck` と `pnpm build` が通る。

## 成功指標

ユーザー指標:

- 比較検討中も検索結果の地図・店舗一覧の文脈を離れずに比較できる。
- 比較を開く・閉じるのコストが下がり、候補の出し入れと比較を往復しやすくなる。

開発指標:

- `/compare` ルートと関連コンポーネント・状態（`finalStoreId` 系）が削除され、比較に関する状態・コンポーネントが `/results` 配下に一本化される。
- 未使用コード（`FinalStorePanel`、`buildFinalStoreMessage`、`buildGoogleMapsUrl` など）が残らない。
- `pnpm run typecheck` と `pnpm build` が通る。

## モック UI 方針

このサイクルは `/results` に新しい相互作用パターン（比較サイドパネルによる地図エリア上書き）を導入するため、`UNIT_OF_WORK.md` を生成する前に軽量な mock UI を実装して確認する。

mock UI は本実装の足場として残す想定で、`/results` の比較トレイから既存の `CompareTable` を地図エリア上書きパネルとして表示できるようにする。「この店に決める」ボタンと `FinalStorePanel` は最初から含めない。mock UI 確認後、`/compare` ルート・関連コンポーネント・`finalStoreId` 系状態の削除を含めて `UNIT_OF_WORK.md` を生成する。
