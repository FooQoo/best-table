# 計画

## 現在の目的

検索結果画面と最終候補パネルで、プレースホルダーではなく実際の地図を表示できる状態にする。

現状は Gemini の Google マップによるグラウンディング応答に座標が含まれないため、`Restaurant.location` が常に `null` になり、`/results` の MAP と `/compare` の最終候補パネルが地図として機能していない。このサイクルでは、`placeId` から座標・住所・代表写真を解決する境界を追加し、地図 UI を実データ形状で動かせるようにする。

課金と初期表示負荷を抑えるため、検索結果は一度に全件取得しない。初回は10件だけ取得し、ユーザーが店舗一覧の最下部までスクロールしたときに追加10件を取得する。

あわせて、地図実装を安定して開発・確認するため、`pnpm dev:mock` で AI 呼び出しと Places 呼び出しを止め、座標・住所・代表写真付きの固定モック検索結果を返す開発導線を用意する。

## 実装範囲

画面・パスの実装対象:

- MAP 付き検索結果: `/results`
- 10件単位の検索結果取得と追加読み込み: `/results`
- 最終候補の地図表示: `/compare` の最終候補パネル
- 検索 resource route: `/api/restaurants/search`
- 開発用 mock mode: `pnpm dev:mock`

実装対象外:

- 実予約
- 実在庫連携
- 地図上での予約導線・空席表示
- モバイル専用の地図 UX（ボトムシート、リスト/地図切替など）
- 店舗 DB や本番検索エンジンの新規構築
- 地図 UI からの再検索、範囲検索、ドラッグ検索
- 11件目以降の自動先読み（追加取得は一覧最下部へのスクロールをトリガーにする）

## プロダクトフロー計画

1. ヒアリング入力: `/hearing`
2. 検索結果を10件取得: `/api/restaurants/search`
3. 座標・住所・代表写真付き候補を保存: `bookingAtom.restaurants`
4. 左側に店舗一覧、右側に地図を表示: `/results`
5. 店舗一覧の最下部までスクロールしたら追加10件を取得して追記
6. 店舗カードと地図マーカーを連動
7. 比較候補を選ぶ: `/results`
8. 比較画面で最終候補を選ぶ: `/compare`
9. 最終候補パネルに選択店舗の地図を表示

## マイルストーン1: 地図に必要な店舗データ

目的:

実検索結果とモック検索結果のどちらでも、同じ `Restaurant.location` / `address` / `photoUrl` を使って地図と店舗カードを描画できるようにする。

タスク:

- `placeId` から座標・住所・代表写真を解決するサーバー側境界を追加する。
- Gemini グラウンディングで取得した候補に対し、Places API で `Restaurant.location` / `address` / `photoUrl` を埋める。
- Place Details Essentials で `location` と同じ SKU に収まる `formattedAddress` / `shortFormattedAddress` / `addressComponents` / `types` / `viewport` / `plusCode` は、必要なものを同じ FieldMask で同時取得する。
- `displayName` / `rating` / `userRatingCount` / `regularOpeningHours` / `priceRange` / `reviews` など、上位 SKU になるフィールドはこのサイクルでは同時取得しない。
- 座標解決に失敗した店舗は `location: null` のまま扱い、架空の座標を作らない。
- 住所・写真取得に失敗した店舗は既存のフォールバック表示に戻す。
- `app/mocks/` に座標・住所・代表写真付きの検索結果モックを用意する。
- `pnpm dev:mock` を追加し、`MODE=mock` では `/api/restaurants/search` が Gemini や Places を呼ばずモックを返す。

受け入れ条件:

- `pnpm dev:mock` で起動した場合、検索結果に座標・住所・代表写真付きの店舗が返る。
- `pnpm dev` は `MODE=real`、`pnpm dev:mock` は `MODE=mock` で起動する。
- mock mode では Gemini の候補探索・構造化評価、Places の座標・住所・写真取得を呼ばない。
- 通常 mode では `placeId` がある店舗について座標・住所・代表写真の解決を試みる。
- 通常 mode の Place Details は `location,formattedAddress,shortFormattedAddress,types,viewport,plusCode,photos` を基本候補とし、同じ SKU に収まるフィールドだけを追加する。
- 通常 mode の外部 API 呼び出しは10件単位に制限される。
- 座標が取れない店舗は一覧・比較には表示されるが、地図マーカーにはならない。
- 住所・写真が取れない店舗は「情報なし」またはプレースホルダーで表示される。

## マイルストーン2: 10件単位の追加取得

目的:

検索結果を一度に30件取得せず、初回10件とスクロール時の追加10件に分けて、Places API の課金と初期表示負荷を抑える。

タスク:

- `/api/restaurants/search` のリクエスト/レスポンスに、取得範囲（例: `limit`, `offset` または `pageToken` 相当）を追加する。
- 初回表示では10件だけ取得する。
- 初回検索待ちは専用ロード画面にせず、店舗一覧カラムにレストランカード形状のスケルトン UI を表示する。
- 店舗一覧の最下部に到達したら、追加10件を取得して既存 `Restaurant[]` に追記する。
- 追加取得中、取得済み店舗の閲覧・比較操作はブロックしない。
- 追加取得中は一覧末尾に追加分のスケルトンを出す。
- 追加取得が失敗した場合は、既存結果を維持したまま再試行導線を表示する。

受け入れ条件:

- `/results` 初期表示では最大10件だけ表示される。
- 初回検索待ちで専用ロード画面に切り替わらず、店舗カードのスケルトンアニメーションが表示される。
- 一覧最下部までスクロールすると追加10件が追記される。
- 追加取得中は取得済み店舗の下にスケルトンが表示される。
- 追加取得中も、既存店舗の比較追加・詳細遷移・地図操作が使える。
- 追加取得に失敗しても、取得済み店舗は消えない。

## マイルストーン3: 検索結果地図

目的:

`/results` の右側 MAP を、プレースホルダーではなく操作可能な地図として表示する。

タスク:

- `@vis.gl/react-google-maps` を使って `ResultsMap` を実地図コンポーネントへ置き換える。
- `Restaurant.location` を持つ店舗をマーカーとして表示する。
- 地図の初期表示範囲を、店舗群またはヒアリングのエリア座標に合わせる。
- マーカークリックで対応する店舗カードを強調する。
- 店舗カード側の選択またはフォーカスで対応マーカーを強調する。
- Google Maps JavaScript API の読み込み失敗時は、地図部分だけエラー表示にする。

受け入れ条件:

- `/results` の右側に実際の地図が表示される。
- 座標付き店舗がマーカーとして表示される。
- マーカーと店舗カードの対応が画面上で分かる。
- 地図が読み込めない場合でも、店舗一覧、比較追加、比較画面への遷移は使える。
- デスクトップ左右分割を主対象とし、モバイル幅では少なくとも表示が破綻しない。

## マイルストーン4: 最終候補地図

目的:

比較後に選んだ最終候補について、予約前に位置を確認できる地図を表示する。

タスク:

- `/compare` の `FinalStorePanel` 内の地図プレースホルダーを実地図に置き換える。
- 最終候補の `Restaurant.location` がある場合、その1店舗を中心にマーカー表示する。
- `location` がない場合は、地図を出さずに「地図情報なし」を明示する。
- 最終候補の説明文面や予約前確認事項を邪魔しない寸法にする。

受け入れ条件:

- 最終候補を選ぶと、その店舗の地図が最終候補パネルに表示される。
- 座標がない店舗では、架空の位置を表示せず、地図情報がないことが分かる。
- 既存の「この店舗を選んだ理由」「予約前の確認事項」は読みやすいまま維持される。

## 成功指標

ユーザー指標:

- 検索結果で候補の位置関係を把握できる。
- 初回表示が10件に絞られ、待ち時間中も店舗一覧のスケルトンで結果が入る場所を理解できる。
- 必要に応じて追加10件を自然に読み込める。
- 最終候補の予約前確認で、場所に関する不安が減る。
- 店舗カードと地図マーカーの対応が直感的に分かる。

開発指標:

- `pnpm dev:mock` で AI API キー、Places API キー、Gemini の応答に依存せず、地図 UI を確認できる。
- mock mode と通常 mode が同じ `Restaurant[]` の形で `/results` に渡る。
- Places API 呼び出しは10件単位で発生し、上限・キャッシュ・失敗時挙動をテストできる。
- `pnpm run typecheck` と `pnpm build` が通る。

## モック UI 方針

このサイクルは新しいユーザー可視 UI（地図表示、マーカー連動、最終候補地図、追加読み込み）を含むため、`UNIT_OF_WORK.md` を生成する前に軽量な mock UI を実装して確認する。

mock UI は本実装の足場として残す想定で、`pnpm dev:mock` による座標・住所・代表写真付きモック検索結果を使う。AI 呼び出しや実座標解決に依存せず、`/results` と `/compare` の地図表示・カード連動・最終候補地図・10件単位の追加読み込みを先に検証する。
