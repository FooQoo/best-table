# セキュリティ

## セキュリティ方針

Best Table はレストランの意思決定支援プロダクトです。店舗探索・評価・地図コンテキスト AI チャットは実際の Gemini API（本番の外部 AI サービス）を呼び出しており、単なるモックデータではありません。一方で予約・在庫・店舗データベースは実接続しておらず（`docs/DESIGN.md` の実装対象外）、この境界を越えて実予約や実在庫を扱う本番運用では、ユーザーの希望条件、予約文脈、店舗データ、口コミ、AI プロンプトをより広く扱う可能性があります。

セキュリティ上の優先事項:

- ユーザーが入力した会食文脈を保護する。
- 店舗情報や口コミを経由したプロンプトインジェクションを防ぐ。
- AI 出力は根拠カテゴリに基づき、権威的に断定しすぎない。
- 予約・在庫の真実と生成文を分離する。

## センシティブデータ

センシティブになり得るユーザー入力:

- 会食相手の種別
- 「その他」の相手に関する自由入力
- 避けたい条件
- 予算
- 日時
- 人数
- 選択した店舗
- 地図コンテキスト AI チャットの質問内容

センシティブになり得る事業データ:

- 予約空席
- 提携店舗メタデータ
- 内部スコアリング特徴量
- AI 評価キャッシュ

## プロンプトインジェクションのリスク

店舗説明、口コミ、メニュー、ユーザー自由入力は、すべて信頼できない入力として扱う。

リスク:

- 口コミがモデルにルール無視を指示する。
- 店舗説明に宣伝的または攻撃的な文章が含まれる。
- ユーザー自由入力が隠しプロンプトやシステムデータの抽出を試みる。

対策:

- 店舗情報や口コミは指示ではなくデータとして扱う。
- システム指示と情報源の抜粋を分離した構造化プロンプトを使う。
- 長い生テキストよりも、抽出済みの根拠フィールドを優先してプロンプトに渡す。
- 生成出力で引用する情報源テキストは短く制限する。
- 隠しプロンプト、モデル設定、内部スコアリング式を露出しない。

## 出力安全性

AI 出力で行ってはいけないこと:

- 店舗を中傷する。
- 裏付けのない否定的主張を事実として出す。
- 適性、空席、予約成功を保証する。
- 個室の有無や空き状況を捏造する。
- ユーザーが意図していない非公開の生テキストを含める。

AI 出力で行うこと:

- 推論には慎重な言葉を使う。
- 根拠カテゴリに基づいて理由を示す。
- 必要に応じて「予約前に確認すべき点」を含める。
- ビジネス用途に合う簡潔な文体にする。

## アクセス制御の考慮事項

本番では次を守る。

- ユーザーセッションごとに検索文脈と選択店舗を分離する。
- 共有リンクは、明示的に意図した非公開メモだけを公開する。
- 管理者専用の店舗情報やスコアリングメタデータをクライアントに送らない。
- 予約・空席 API は信頼できるサーバー側コードから呼ぶ。

## データ保持の考慮事項

プロトタイプ:

- ローカルのモック状態のみ。
- デフォルトでは個人データを永続化しない。

本番:

- 自由入力の会食文脈の保持を最小化する。
- 必要がない限り、センシティブなゲスト詳細を保存しない。
- ログには不要な個人文脈を含めない。

## クライアント側セキュリティ

フロントエンドのルール:

- テキストは HTML ではなくテキストとして描画する。
- AI 出力を生 HTML として注入しない。
- 自由入力の長さを検証する。
- 生成AI API の認証情報はサーバー側に置く。
- 内部スコアリングプロンプトをブラウザに送らない。
- 地図表示用の Google Maps JavaScript API キーは、Gemini・Places 用のサーバー専用キーとは別に発行し、HTTP リファラー制限をかけたうえでクライアントに渡す。
- `placeId` から座標・住所・代表写真を解決する API キーはサーバー側に置き、ブラウザには渡さない。

## サーバー側と AI のセキュリティ

サーバー側のルール:

- 検索・AI リクエスト入力をすべて検証する。
- mock mode は開発用環境変数でのみ有効化し、本番で意図せず AI 呼び出しを迂回しない。
- AI エンドポイントにレート制限をかける。Google マップによるグラウンディングなど、外部データを参照する呼び出しにも1検索あたりの回数上限を設ける。
- 地図コンテキスト AI チャットでは、クライアントから送られた `Restaurant[]` を信頼せず、サーバー側で形状・件数・質問文長を検証する。
- プロンプトサイズと情報源ドキュメント長に上限を設ける。
- ログをサニタイズする。
- 現在在庫の取得処理と AI 生成を分離する。
- 可能な限り型付きの構造化データを返す。

## セキュリティの受け入れ条件

- ユーザー自由入力がシステム指示を変更できない。
- AI 出力が安全でない HTML として描画されない。
- 生成まとめが隠しプロンプトや無関係な非公開文脈を露出しない。
- 予約に関する真実を AI だけで推論しない。
- センシティブな空席情報や提携メタデータを、意図なくクライアントに露出しない。
