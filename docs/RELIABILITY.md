# 信頼性

## 信頼性の目的

Best Table は、重要な会食判断を支援するプロダクトです。信頼性とは、不確実性を下げる一方で、システムが知らないことまで知っているように見せないことです。

信頼性は次の3つで考える。

- プロダクトの信頼性: データが不完全でもフローが使える。
- データの信頼性: 情報源に基づく事実と推論を分ける。
- AI の信頼性: 生成文が根拠に基づき、慎重で、行動につながる。

## 信頼できる情報源の境界

AI は要約や比較に使えるが、AI 出力そのものを一次情報として扱ってはいけない。

厳守する境界:

- 空席や予約状態は、在庫・予約システムから取得する。
- 店舗事実は、店舗データ、メニュー、席情報、写真、口コミ、アクセス説明などに基づく。
- AI は適性を推論してよいが、不確実性と根拠カテゴリを示す。
- 根拠が不足している場合は、補完せず信頼度を下げる。

## AI 回答の信頼性

AI 回答で行うこと:

- 断定しすぎない。
- 根拠カテゴリを示す。
- 懸念点と注意事項を含める。
- 予約前に確認すべき次アクションを示す。
- 最終判断はユーザーに残す。

AI 回答で行わないこと:

- 適性を保証する。
- 予約成立を保証する。
- 席、メニュー、接客、雰囲気を捏造する。
- 現在情報の取得元なしで、現在の空席を主張する。
- 不確実性を自信のある言葉で隠す。

マッチ度（`Restaurant.matchTier`）は AI 生成ではなく決定的な計算だが、同じ原則が適用される。判定に使える AI 評価済みフィールドが1つも無い場合（AI評価が未到達、または全て不明）は、`null`（評価未生成）のままにし、根拠のない段階を捏造しない（算出方法は `docs/MODEL.md`「コンテキスト2」を参照）。

## 段階的表示

店舗候補そのものが Places API による施設検索と Gemini の AI 評価によって生成されるため（`docs/ARCHITECTURE.md` の「検索・評価型」）、検索は通常の店舗データベース検索と同等の速度では表示できない。この前提のもとで、以下の方針を取る。

- 検索結果のキャッシュは現時点で実装していないため、同一条件の再検索でも毎回この段階的表示を経る。
- `/api/restaurants/search/stream` は施設検索（Places API）の結果を NDJSON で先に返し、店舗一覧はAI評価を待たずに一覧へ表示される。AI評価（マッチ度含む）は店舗ごとに後から反映され、一覧はカードが増えるのではなく既存カードの内容が更新される（並び順は施設検索が返した順のまま固定し、評価の到着順では並び替えない）。専用ロード画面に切り替えず、Places API による候補探索中だけ店舗カード形状のスケルトン UI を表示する。AI評価中はスケルトンを出さず、現在のフェーズと完了件数（AI評価が完了した件数）を短いステータス文で示し、無反応に見えないようにする。実在しない店舗名や架空の店舗情報では埋めない。
- 生成に時間がかかる、またはタイムアウトする場合も、比較・戻る等のナビゲーションは操作可能な状態に保つ。
- `/results` の地図コンテキスト AI チャット（`streamText`）は、この検索結果生成とは別物であり、ストリーミングで逐次表示してよい。チャットが失敗しても、表示済み店舗一覧、地図、比較操作は維持する。

## 応答速度

AI 応答速度を上げるため、次の方針を取る。

- Gemini のモデル ID は `app/server/clients/gemini-models.ts` に集約する。検索結果の構造化評価（`GEMINI_EVALUATION_MODEL_ID`）は `gemini-3.5-flash` で判断品質を優先し、地図コンテキスト AI チャット（回答本文・おすすめ質問、`GEMINI_CHAT_MODEL_ID`）は `gemini-3.1-flash-lite` で低レイテンシ・低コストを優先する。両方とも Flash 系だが、用途に応じて別モデルを使い分ける。
- 地図コンテキスト AI チャットは `streamText` のままにし、全文生成完了を待たずに逐次表示する。
- チャット回答は `maxOutputTokens` と prompt 上の行数制限で短く保つ。長文回答より、比較・懸念・次アクションに絞った回答を優先する。
- 回答後のおすすめ質問生成は `timeout` と `maxRetries: 0` を設定し、遅い場合は deterministic な候補に即フォールバックする。
- クライアントが回答生成を中断した場合は `request.signal` を AI SDK に渡し、不要な生成を続けない。

## 想定される失敗

想定される失敗:

- 個室情報が欠けている。
- 口コミが少ない。
- 写真が古い、またはカテゴリが不明確。
- 店舗写真の Place ID 解決や取得に失敗する、または対象の写真が存在しない。
- `placeId` からの座標解決に失敗する、または対象店舗の座標が取得できない。
- 10件単位の追加取得が失敗する、または途中で打ち切られる。
- Google Maps JavaScript API の読み込みに失敗する。
- 現在在庫を取得できない。
- 予算やメニュー情報に不整合がある。
- AI がタイムアウトする、または一部しか返らない。
- 地図コンテキスト AI チャットがタイムアウトする、または回答生成に失敗する。
- 店舗候補探索で使う Places API の施設検索が失敗する、または該当0件になる。

必要な振る舞い:

- 利用可能な基本データを表示する。
- 不明な項目を明確に表示する。
- 根拠のない自信を生成しない。
- 「予約前に個室可否を確認」などの次アクションを提示する。
- 比較とナビゲーションは使える状態に保つ。
- 座標が取れない店舗は地図マーカーを表示せず、一覧・比較の表示は維持する。
- 追加取得に失敗しても取得済み10件は消さず、再試行できる状態にする。
- 候補探索待ちのスケルトンは実店舗情報として誤認されない見た目にする。AI評価待ちをスケルトンで表現しない。
- 地図コンテキスト AI チャットに失敗しても、地図・店舗一覧・比較候補は消さず、チャットパネル内だけで失敗を示す。
- 送客リンクの URL は店舗名から組み立てる一休.com検索URL（一休）と `placeId` ベースの URL 構築（Google Maps）に限定し、AI 出力や自由入力から送客 URL を生成・補完しない。

## データ鮮度

鮮度の影響が大きい項目:

- 空席
- 営業時間
- コース料金
- メニュー詳細
- 席の空き状況
- 定休日

鮮度の影響が比較的小さい項目:

- 所在地
- 一般的なジャンル
- 過去の口コミ傾向
- アクセス説明

UI は、鮮度が必要な情報を最近のデータなしに断定しない。

## URL query state の復元性

`mvp-cycle-5` では、トップ・ヒアリング・検索結果で指定する検索・会食条件を URL query state として保持する。これにより、リロード、直接 URL アクセス、共有リンク、ブラウザの戻る/進むでも、検索結果の前提条件を復元できるようにする。

必要な振る舞い:

- URL query が空の場合は、明示的な初期値にフォールバックする。
- URL query の一部が壊れている場合は、固定語彙・型・上限に合わせて正規化し、検索 API や AI prompt に不正値を渡さない。
- URL query の検索条件が変わった場合は、取得済み店舗、比較候補などの一時状態を条件に合わせてクリアし、古い条件の結果を新しい条件の結果として見せない。
- 共有 URL で再現する対象は検索条件までとし、比較候補、追加検索結果、詳細パネル、地図凡例などの画面内表示状態は再現を保証しない。

## 信頼性の受け入れ条件

- キャッシュ済みの検索条件では、結果表示が AI 生成にブロックされない。
- 初回検索など生成を待つ間も、無反応に見えず、生成中であることが明確に分かる。
- AI データがない店舗でも閲覧・比較できる。
- 座標データがない店舗でも閲覧・比較できる。
- 追加取得中でも取得済み店舗の閲覧・比較ができる。
- AI 文言が懸念点と根拠カテゴリを示している。
- 実ソースなしに確定在庫を主張しない。
- URL query から復元した条件で `/results` を直接開いても、初期値ではなく URL の条件で検索される。
