# デザイン

## 体験の方向性

Best Table は、汎用的なレストラン探索アプリではなく、予約判断のリスクを下げるための業務寄りのツールとして感じられるべきです。

デザインで伝えるべき印象:

- 信頼感
- 落ち着き
- ビジネス用途への適合
- 素早い比較
- 予約前の不安の少なさ

ビジュアル方向性は、ネイビー、ゴールド、オフホワイト、インク系の落ち着いた配色です。装飾的ではなく、上質で実務的に見えることを優先します。

## 全体フロー

1. ヒアリング入力
2. MAP/検索結果
3. AI 評価付き店舗カード
4. 複数店舗の比較（検索結果画面内のサイドパネル）
5. 地図上の店舗群に対する AI 相談

## プロダクト仕様サマリー

このセクションは、デザイン検討時に参照するプロダクトの使用面・仕様面の要点を集約する。

### 中心コンセプト

> 大切な食事の場を、失敗しない選択に変えるAI比較サービス。

Best Table は、MAP 検索で見つけたレストラン候補を、接待・会食の文脈に合わせて AI が比較・要約し、予約判断を支援するサービスです。

### プロダクトのゴール

> Best Table のゴールは、接待・会食の店選びで「比較 → 納得」までを担い、納得した候補を一休.comの検索結果ページへ送客すること。予約・空席確認は一休.com側で完結させ、Best Table は予約機能を持たない。

- 一休掲載店マスタとの店舗同定（照合）は行わない。送客リンクは店舗名を検索語にした一休.comの検索結果ページ URL（`https://restaurant.ikyu.com/search?term=<店舗名>`）を、全店舗に一律で表示する。
- 送客リンク（「一休.comで空席を確認」、既存の「Google Mapで空席・予約を確認」）は、店舗詳細パネルと比較サイドパネルに置く外部確認導線であり、アプリ内の確定・決定操作ではない（`mvp-cycle-4` で撤回した最終候補選択は復活させない）。
- 「空席・予約成立を断定しない」ガードレールは維持し、空席・予約の確認先として一休.com / Google Mapを案内する。
- 北極星指標は「検索セッションあたりの送客クリック率」（`mvp-cycle-6` では指標定義のみ。計測は未実装）。

### 利用シーン

主に、店選びの失敗が目立ちやすく、説明責任や心理的コストが高い会食で使う。

- 重要顧客との接待
- 初回取引先との商談前後の会食
- 社外役員・上役クラスとの食事
- 社内上司・幹部との会食
- お礼・懇親の食事
- 関係を深めたい相手との会食

### 主要ユーザー

- 接待・会食の店選びを任された若手社員
- 役員や上司の会食を手配する秘書・アシスタント
- 顧客との食事を計画する営業担当
- 印象を損ねられない食事の場を準備する人

### ユーザーが解決したいこと

- 口コミ、写真、席情報、メニューを読み込む時間を減らしたい。
- 似た候補を同じ軸で比較したい。
- 相手や会の目的に合っているかを判断したい。
- 個室、騒がしさ、雰囲気、アクセス、予算などの懸念を予約前に把握したい。
- 上司、秘書、同席者に説明できる選定理由がほしい。

### プロトタイプで示す体験

- 検索入口は従来型の MAP/条件検索として維持する。
- AI は候補発見後の比較、要約、懸念抽出、説明、質問応答に使う。
- 会食文脈をヒアリングし、マッチ度や比較軸に反映する。
- 複数店舗を短時間で比較し、判断材料をそろえられるようにする。

### 現在の実装対象

- トップ / 検索入口
- ヒアリング画面
- 検索結果（Places API による施設検索 + Gemini 構造化評価による実データ。座標・住所・代表写真は施設検索のレスポンスから直接取得し、Google Maps JavaScript API で MAP 表示する）
- AI 評価付き店舗カード
- 検索結果内の店舗詳細パネル
- 一休.comへの送客リンク（店舗詳細パネル・比較サイドパネルの「一休.comで空席を確認」。店舗名を検索語にした一休.com検索結果ページへの送客。`mvp-cycle-6`）
- 地図上の店舗群に対する AI チャット
- 比較サイドパネル（`/results` 内、地図エリアを上書き表示）

### 現在の実装対象外

- 実予約（予約は送客先の一休.com / Google Map 側で完結する。Best Table は今後も予約機能を持たない）
- 実在庫連携（空席情報は常に「確認できない」という留保表現で扱い、確認先として送客先を案内する。`docs/RELIABILITY.md`）
- 実店舗データベース・本番検索エンジン（自前の店舗 DB や検索インフラは持たず、Places API (New) の Text Search に候補探索を任せている）
- 一休掲載店マスタとの店舗同定（照合）。送客は店舗名検索のみとし、一休掲載の有無は判定しない
- 送客クリックの計測実装（北極星指標の定義のみ）
- 最終候補選択・アプリ内予約導線ページへの遷移（`mvp-cycle-4` で撤回。比較は複数店舗を見比べる機能に限定し、確定・決定という概念は持たない。一休.comへの送客リンクは確定操作ではなく外部確認導線として扱う）

Gemini（Google の本番 AI 推論基盤）自体は UoW-7 で実接続済みで、店舗候補の探索・評価は実際の Gemini API を呼び出している。`mvp-cycle-3` 以降、単一店舗詳細ページの質問応答は廃止し、AI 相談は検索結果地図上の店舗群をコンテキストにしたチャットへ集約する。「実装対象外」なのは店舗データベース・在庫連携・予約であり、AI 推論そのものは対象外ではない。

### AI の役割

AI が行うこと:

- 店舗情報の要約
- 口コミの論点抽出
- 写真や説明文からの雰囲気の言語化
- 懸念点の抽出
- ヒアリング結果に応じた評価軸の調整
- 比較表の生成
- 地図上の表示中店舗群に関する質問応答

AI が行わないこと:

- 構造化検索条件の完全な置き換え
- 空席在庫の確定判断
- 予約成立の保証
- 店舗評価の断定
- 口コミや情報源にない事実の補完

### 画面上のガードレール

- AI ファーストのレストラン検索チャットにしない。
- エリア、日時、人数、予算、個室希望などの構造化条件は正確に扱う。
- 重要な懸念はカードや比較表に直接表示し、ホバーだけに隠さない。
- AI 文言は短く、根拠カテゴリや不確実性が伝わる表現にする。
- 空席、予約成立、裏付けのない店舗事実を断定しない。
- AI コメントが遅い場合でも、基本の検索結果と比較操作は使える状態に保つ。

### 状態・操作の制約

- 検索基本条件: エリア、日付、時刻、人数
- 会食文脈: 相手種別、相手に関する自由入力
- 予算: 下限、上限、自由入力
- 重視条件: 最大3つまで
- 比較候補: 最大5件まで
- 通常の比較フローでは、2件以上の選択を必要とする
- `mvp-cycle-5` では、トップ・ヒアリング・検索結果で指定する検索基本条件、会食文脈、予算、重視条件を URL query state として保持する。リロード、直接 URL アクセス、共有リンクではこの query から条件を復元する。
- URL query で共有・復元する対象は検索条件までとし、比較候補、取得済み店舗、詳細パネル、地図凡例の表示状態、地図の追加検索結果は画面内の一時状態として扱う。

### 用語対応表

トップ・ヒアリング・検索結果・比較の4画面で表記ゆれを起こさないための正式表記。
新しい文言を追加・変更する際はこの表に合わせる。

| 正式表記 | 用途 | 使わない表記 |
|---|---|---|
| 相手／相手種別 | 会食の相手を選ぶステップ、状態名 (`counterpart`) | ご一緒する方、参加者 |
| 重視条件 | 外したくない条件を選ぶステップ、状態名 (`priorities`) | こだわり、優先事項 |
| 比較に追加／比較から外す | 検索結果カードで比較トレイへの出し入れ | 比較へ、選択する |
| 比較する | 検索結果画面で比較サイドパネルを開くボタン | 比較へ進む |
| 一休.comで空席を確認 | 店舗詳細パネル・比較サイドパネルの送客リンク（一休掲載店のみ） | 一休で予約する、予約へ進む、空席を見る |
| Google Mapで空席・予約を確認 | 店舗詳細パネル・比較サイドパネルの Google Maps 場所ページへの送客リンク（全店舗） | Google Mapで開く、地図で見る |

## デザイン原則

- 既存の検索行動を維持する。
- AI 評価は候補比較の場面に置く。
- 利点だけでなく、リスクを先に見えるようにする。
- 説明はスキャンできる短さにする。
- カード、比較表で同じ比較軸を使う。
- 意思決定画面では長い AI 文章を避ける。

## ヒアリング設計

目的:

会食の文脈を最小限入力してもらい、検索結果の説明や比較軸に反映する。


### ステップ1: 基本条件

入力項目:

- エリア
- 日時
- 人数

設計意図:

ここはクラシックな検索条件として扱う。自然文 AI 検索ではなく、構造化された検索基盤で正確に絞り込む。

### ステップ2: 会食の相手

見出し:

> 今回の会食について、少しだけ教えてください

質問:

> どなたとの会食ですか？

選択肢:

- 重要顧客・役員クラスの接待
  - 個室・接客・格式を外したくない
- 初回の取引先・商談前後
  - 堅すぎず、会話しやすい雰囲気にしたい
- 社内上司・幹部との会食
  - 予算感と落ち着きを両立したい
- お礼・懇親の食事
  - 感謝が伝わる、少し特別な店にしたい
- 関係を深めたい相手との会食
  - 肩肘張らず、印象に残る食事にしたい
- その他
  - 自由入力

設計意図:

単なる属性分類ではなく、会食の緊張度や失敗できないポイントを取得する。これにより、後続の評価軸や重み付けを変える。

### ステップ3: 重視条件

見出し:

> この会食で、特に外したくないことは何ですか？

補助文:

> 最大3つまで選択してください。選んだ条件をもとに、検索結果のマッチ度と比較表を調整します。

選択肢:

- 落ち着いて話せる
  - 周囲の音や席間隔を重視します
- 個室・半個室を優先
  - 商談や相談がしやすい席を優先します
- 失礼のない格式感
  - 相手の役職や関係性に合う雰囲気を重視します
- 接客の安定感
  - 口コミ上のサービス評価を重視します
- 駅から迷わず行ける
  - 初めての相手でも到着しやすい店を優先します
- 予算内に収めたい
  - 想定単価から大きく外れない店を優先します
- 苦手・避けたい条件がある
  - 自由入力

相手に応じた補助文:

- 重要顧客・役員クラス: 個室・接客・格式感を強めに見る
- 初回の取引先: 会話しやすさと失礼のない雰囲気を重視
- 社内上司・幹部: 落ち着きと予算感のバランスを重視
- お礼・懇親: 特別感と居心地を重視
- 関係を深めたい相手: 話しやすさと印象に残る雰囲気を重視

## 検索結果画面

基本レイアウト:

- 左側: 店舗一覧
- 右側: MAP
- 上部: 検索条件の要約
- 店舗カード: 基本情報 + AI 評価

検索中の表示:

- 施設検索の結果はAI評価を待たずに先に一覧へ表示し、AI評価（マッチ度含む）は店舗ごとに後から反映される（一括で全件の評価が揃うのを待たない）。一覧の並び順は施設検索が返した順のまま固定し、評価の到着順では並び替えない。
- 検索条件の確認 → 候補探索 → AI 評価、の現在フェーズと完了件数（AI評価が完了した件数）を、短いステータス文で表示する（例:「AIが店舗を評価しています…（3件完了）」）。
- 初回検索待ちは専用ロード画面にせず、Places API による候補探索中だけ店舗カード形状のスケルトンアニメーションを一覧に表示する。AI評価中はスケルトンを出さず、既に表示した店舗カードを後から更新する。

MAP の振る舞い:

- 初回は10件まで表示し、店舗一覧の最下部までスクロールしたら追加10件を読み込む。
- 店舗の `location` がある場合だけ、地図上にマーカーを表示する。
- `location` がない店舗は一覧・比較には残すが、架空のマーカーは作らない。
- マーカークリックで対応する店舗カードを強調し、右側の詳細パネルを表示する。
- 店舗カード側の選択またはフォーカスで対応マーカーを強調する。カードクリックではページ遷移せず、右側の詳細パネルを表示する。
- 店舗カードをクリックして詳細パネルを表示したとき（マーカークリックで詳細パネルを表示したときも同様）は、地図の中心を対応するピンの位置へ移動する（`panTo`）。ホバー・フォーカスによる強調だけでは中心移動しない。
- ユーザーが地図を動かして中心座標が変わった場合、地図上部中央に「このエリアを検索」ボタンを表示する。押下すると日時・人数・予算・重視条件などは維持し、旧エリア名は検索語から外して地図中心座標を優先した候補を追加検索する。既に表示済みの店舗は残し、新しく見つかった店舗だけを末尾に追加する。重複店舗は `placeId`、または `placeId` がない場合は店名・住所で除外する。
- 地図右下に AI チャットボタンを置き、押すと地図エリア内の右側から AI チャットパネルを表示する。
- AI チャットは現在 `/results` に読み込まれている店舗すべてをコンテキストにする。
- AI チャットの初回画面には、地図上の店舗群に対するよくある質問を表示する。
- AI が回答した後は、次のおすすめ質問を4つ表示する。
- 地図読み込みに失敗しても、店舗一覧、比較追加、比較サイドパネルの表示は使える。
- 地図ピンはマッチ度（最高・高・中・低の4段階）で常時色分けする。評価未生成の店舗はピンの色としては「低」と同じ扱いにする（色は分けない）。地図右上に GIS 風の絞り込みパネルを1枠で常時表示する（評価やトレイ追加の到着を待たない）。パネルは「マッチ度」セクション（最高・高・中・低・評価未生成の5行、チェックボックス+色ドット。評価未生成は色ドット無し）と、その下の「比較」セクション（「比較対象」「比較対象外」の2行、チェックボックス）から成る。各行には該当件数を表示し、クリックでチェックのオン/オフができ、オフにした行だけが地図から一時的に隠れる（色分けの全体をオフにするマスタースイッチは無い）。「比較対象外」のチェックを外すと比較トレイに追加済みの店舗のピンだけを、「比較対象」のチェックを外すと未追加の店舗のピンだけを地図から隠す。非表示にした行の店舗カードは、一覧上では残したままグレー表示にする。この絞り込み状態はURL query に含めず画面内の一時状態として扱う。

店舗カードの項目:

- 店舗名
- ジャンル
- エリア
- 予算
- 個室・席
- 代表写真
- 個室有無
- AIによるマッチングポイント要約
- 比較に追加ボタン
- カードクリックで表示する詳細パネル（推奨理由、懸念点、空席状況の留保表現、基本的な確認観点、「一休.comで空席を確認」リンク、「Google Mapで空席・予約を確認」リンク）
- 追加取得中の状態、追加取得失敗時の再試行導線
- 追加取得中は Places API による候補探索中だけ、既存カードの下に追加分のスケルトンカードを表示する。AI評価中は追加スケルトンを出さない。

AIによるマッチングポイント要約例:

> 個室の記載があり、口コミでも接客の丁寧さに言及が多い。重要顧客との会食では安心度が高い一方、駅から少し歩く点は事前案内したい。

## 比較サイドパネル

目的:

同じ判断軸で店舗を見比べられるようにする。最終候補の確定や予約導線への遷移は扱わない。

表示位置:

- `/results` の比較トレイの「比較する」ボタンはトグルで、押すたびに開閉する（開いている間はボタンが「比較を閉じる」に変わる）。専用の閉じるボタン（×）は置かない。
- 地図コンテキスト AI チャットと同じく、`/results` の地図エリア内に重ねて表示する。
- AI チャットの右スライド幅パネルとは異なり、比較表は店舗数に応じた幅が必要なため、地図エリアを隙間なく完全に覆う。
- 左の店舗一覧は覆わず、比較トレイのボタンをもう一度押すと地図表示に戻る。
- パネルを開いた状態でも、店舗一覧の閲覧・比較追加・比較から外す操作は維持する。

表示項目:

- 店舗名
- ジャンル
- エリア
- 予算
- 個室・席
- 代表写真
- 個室有無
- AIによるマッチングポイント要約
- ジャンルの右に「Google Mapで空席・予約を確認」リンク（Google Maps の場所ページへの送客導線。場所ページには店舗によって予約リンクが表示される）
- 各店舗列に「一休.comで空席を確認」リンク（店舗名を検索語にした一休.com検索結果ページへの送客導線。新しいタブで開く。全店舗に一律で表示する）
- 比較軸（マッチ度・個室・静かさ・格式・接客・アクセス・予算目安・懸念点）と、相手種別に応じた重視表示
- 比較表を画像として保存するボタン（社内共有・上司への説明用。保存失敗時はその場でエラー表示し、他の操作はブロックしない）

比較サイドパネルには、最終候補を選ぶボタンや、選んだ店舗のための専用パネルは置かない。送客リンクは「どの店に納得したか」をアプリに記録する操作ではなく、外部で空席・予約を確認するための導線に留める。

## 地図コンテキスト AI チャット

目的:

ユーザーが検索結果地図上の店舗群を見ながら、どの候補を比較に入れるべきか、どの懸念を先に確認すべきかを相談できるようにする。

表示位置:

- `/results` の地図右下に AI チャットボタンを置く。
- チャットパネルは地図エリア内の右側からスライド表示する。
- パネルは店舗一覧を覆わず、地図上に重ねる。
- 閉じる操作を用意し、地図表示へ戻れるようにする。

質問例:

- この条件なら、まず比較に入れるべき店はどれですか？
- 役員クラスとの会食で避けた方がよさそうな懸念はありますか？
- 個室・落ち着き・アクセスのバランスが良い候補はどれですか？
- 予算内に収まりやすそうな候補を優先すると、どの順番で見るべきですか？
- 初回取引先との会食なら、堅すぎない候補はどれですか？
- いま表示されている店舗の中で、比較に入れる3件を選ぶならどれですか？

初回画面:

- 質問前は、上記のようなよくある質問を表示する。
- よくある質問は、表示中店舗群を前提にした横断相談に限定する。
- 空席確定、予約成立、未取得の口コミ本文やメニュー詳細を前提にした質問は初回候補にしない。

回答後:

- 回答の下に、次のおすすめ質問を4つ表示する。
- おすすめ質問は、直前の質問・回答、相手種別、重視条件、表示中店舗群を踏まえたものにする。
- おすすめ質問を押すと、そのまま次の質問として送信できる。

回答方針:

- 断定しすぎない。
- 現在 `/results` に読み込まれている店舗情報とヒアリング条件だけを根拠にする。
- 根拠となる情報種別を示す。
- 懸念点も明示する。
- 予約前に必要な次アクションを提示する。
- 空席、予約成立、未取得の口コミ・メニュー本文を断定しない。
